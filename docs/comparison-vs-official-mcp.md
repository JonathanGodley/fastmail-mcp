# This server vs. the official Fastmail MCP — which to use

> **What this is.** A comparison of two complete Fastmail MCP servers to answer one question: **which should you use?** "This server" = `JonathanGodley/fastmail-mcp` (v1.9.4-fork.7), the self-hosted MCP this repo builds — judged as a whole tool, not on the features that distinguish it from the upstream it was forked from (that lineage doesn't bear on which server to run).
>
> **Freshness header.** Probed **2026-06-25**. Official server: **`Fastmail` v1.0.0**, **29 tools**, protocol `2025-06-18`, remote streamable-HTTP/SSE. This server: **v1.9.4-fork.7** (commit `9f12d35`), local stdio, 40 tools. **Re-probe trigger:** this snapshot is valid as of official v1.0.0 / this server v1.9.4-fork.7 — re-run the audit if either advances, since tool counts, schemas, and token weights can all move.
>
> This is a **dated competitive snapshot**, not a timeless design model. It is deliberately distinct from the `docs/*.md` rationale files (`email-bodies.md`, `security-model.md`, `conventions.md`), which capture durable platform/codebase facts. Don't treat the numbers here as permanent or "fix" this file to match those — it ages by design and should be re-derived, not maintained.
>
> **Evidence grades used below.** Every comparative claim is tagged:
> - **probe-backed** — measured live, read-only, this session (token weights, id-equality, parity smoke).
> - **code-backed** — read from this repo's source (`src/index.ts`, `src/email-formatter.ts`).
> - **schema-inferred (black-box)** — read from the official server's published tool schema/description only; no source, not executed (every official *mutating* tool falls here — we never called one).
>
> **Measurement basis (so the magnitude isn't overread).** Token figures use a local `gpt-tokenizer` o200k_base tokenizer (v2.9.0) — a **GPT-family tokenizer used as a proxy**; the host model (Claude) tokenizes differently, *especially* whitespace, so **raw bytes are the primary, tokenizer-independent evidence** and every directional claim below also holds in bytes. The corpus is one account, one snapshot: a 25-email list page (24 survived the cross-server join), 3 single emails, 1 thread, and a calendar window. Treat directions as solid and exact magnitudes as indicative.
>
> All example payloads in this document are **fabricated**. No real account data, tokens, or addresses appear here.

## 1. Which should you use?

This compares the two servers as **complete tools** — this server judged on everything it does, not on the features that distinguish it from the upstream it was forked from (that lineage is irrelevant to "which should I use"). Tokens are a footnote here, not the verdict (see the note at the end of this section).

**Short answer: there is no forced winner — they optimize for different things. If you'd rather not maintain anything and want the official's broader feature set (Notes, contacts-write, forward, RSVP) now, the official is the better choice. If you value running and controlling your own tooling, and need its exclusives (local download, raw JMAP, structured search, the full draft lifecycle), the self-hosted server is the better fit — with the official as a complement to pull features from or run alongside.** It's a genuine tradeoff; here are the axes.

Both are complete Fastmail MCP servers covering the core job: read/search mail, drafts and replies, send, contacts, calendar. They differ on what surrounds that core.

**Where this server is the better tool to use:**
- **It's self-hosted and yours to change.** Local, in-process, fully under the operator's control, and extensible to anything public JMAP allows. The official is a fixed remote product you can't change or see into — any gap this server has can *likely* be closed in code where public JMAP reaches it (some may not — Notes in particular may be un-portable, §5); the official's gaps you cannot close at all.
- **Attachment download to local disk** (with path confinement). The official has *no* attachment-download tool at all — it uploads but never pulls a file down (schema-inferred).
- **Raw JMAP access** (`raw:true`) and **structured search** (`search_emails`' dedicated from/to/cc/bcc/subject/date/mailbox fields). The official is Gmail-string-only and never exposes the underlying JMAP.
- **A richer draft/compose workflow** — create, *edit*, and send-a-saved-draft, plus sanitized reply quoting and a draft-safe reply default. The official can create a draft but cannot edit it or send a saved one at all (§3). (It does have two compose advantages of its own: `forward`, which this server lacks, and a markdown body that's easier to drive.)
- **You depend on Fastmail's mature JMAP API, not a new product layer.** Both servers go through Fastmail — neither keeps your data local, and every call is a network round-trip either way. The difference is *which* Fastmail surface you rely on: this server rides **JMAP**, Fastmail's long-standing foundational API (the same one their own clients use), while the official rides their **brand-new MCP product** (v1.0.0), which can be rescoped, rate-limited, or changed as a product decision you can't inspect — and the tool/transformation layer in between is code you control rather than a black box.

**Where the official is the better tool to use:**
- **Broader out of the box:** Notes, memos, contacts write, forward, RSVP, calendar invitations (iMIP) — things this server doesn't do yet (schema-inferred).
- **First-party maintained:** it improves on Fastmail's effort, not yours, and won't bit-rot if you stop touching it.
- **Leaner default read output**, carrying slightly more status detail per message (probe-backed).
- **Richer reads on threads and calendars** (probe-backed): `read_thread` returns the conversation's bodies in one call (this server needs a `get_email`×N fan-out), and `search_events` expands recurring events into per-occurrence rows with detail inline (this server lists sparse masters). Both are real read-side capability edges, not token artifacts.
- **Interactive widget UX** (compose/confirm dialogs) if your MCP client renders them.

**An honest scoreboard, so §1 doesn't flatter the home team.** On the *shared* capabilities — where both servers try to do the same job — the **official wins more head-to-head rows** (read-thread, calendar-read, labels, archive, compose-body, string-search; several probe-backed — see the Appendix matrix). This server's case for being the better tool to use therefore rests **less on out-executing the official where they overlap** and **more on** (a) capabilities the official lacks entirely — local attachment download, raw JMAP, `search_emails`' structured field composition, and the full create→edit→send draft lifecycle — and (b) the meta-advantage of being yours to control and extend. Put plainly: if your needs sit inside the shared core, the official is at least as good and often richer; this server pulls ahead precisely where you need one of its exclusives or its malleability. **The two standout keep-reasons today are the faithful draft lifecycle (create→edit→send, which the official can't do — §3) and local attachment handling (download to disk, which the official has no tool for at all).** Those rest on this server's raw-JMAP access and so can't be reproduced by a layer over the official; the official's wins, by contrast (richer thread/calendar reads, the breadth tools), *can* be adopted — either built here or, better for the expensive ones, forwarded via the federated surface (§5, Option 4). If you keep developing this server, the natural split is to hold the draft + attachment core native and pull the rest from the official rather than reinvent it.

**The deciding axis is control + fit vs. breadth + zero-maintenance** — not tokens:
- If this server already does your core mail work and you value owning and shaping your tooling, **it's the better one to use**, and anything it lacks can be added in code (or obtained by running the official alongside).
- If you'd rather not maintain anything and want the official's exclusive features immediately, **the official is the better one to use** — accepting the loss of local download, raw JMAP, structured search, and control.
- **Running both** is a legitimate third answer: yours for local/raw/structured/download, the official for its exclusives. The cost is two overlapping tool sets and a search-dialect gotcha (§5).

One caveat before any switch is *final*, **and it cuts both ways**: this audit was read-only, so the official server's **mutating** side (send/draft/contacts-write/calendar) is inferred from its schemas, never executed — and so is *this* server's (its `edit_draft`/`send_draft`/`bulk_*` behaviour is read from this repo's source this session, not run). Neither server's write path was executed here, so every "X wins on a mutating capability" — in *either* direction — is a code/schema claim, not an observation. A consented write-probe is the honest last check before relying on the official for the things your MCP currently handles.

> **On tokens — a footnote, not the verdict.** This server was built to trim response tokens, and that is its headline difference *from the upstream it forked* — but it is not the axis on which to choose between it and the official server, and an earlier draft of this report wrongly treated it as such. The honest state of the measurement: **as each ships today, this server is heavier, not lighter** (≈322 vs 257 tokens/message; 9,605 vs 6,375 tokens for a 25-mail page) — its founding metric currently *loses* to the official. Compacting the serialization alone (information-preserving) would close much of that gap; going further to ≈204 tokens/message would mean dropping some fields from the *default* view, which is a separate per-field product decision (those fields are real data — see §4), not a given. Either way the gap is small enough that token weight shouldn't pick the server: decide on capability + control. Numbers in §Appendix A2.

Capability detail behind this verdict is §3–§4; the feature-by-feature matrix is the Appendix.

## 2. Decision factors beyond features

- **First-party longevity / maintenance** *(judgment, not a probed finding — but a heavy one).* Official = Fastmail-maintained (first-party, v1.0.0). This server = a single-maintainer codebase (derived from `MadLlama25/fastmail-mcp`). No public release-cadence signal was established this session, so **cadence is not establishable** here — but the bus-factor asymmetry is stark: a company's product vs. one maintainer. The honest trade: **developing this server means one person maintains it against a first-party product that keeps adding features.** One counterweight, in fairness: the official side carries its *own* continuity risk — a first-party remote product can be deprecated, rescoped, rate-limited, or have its token scope changed unilaterally, and you control none of that. So the asymmetry is "one volunteer maintainer vs. one vendor's product roadmap," not strictly risky-vs-safe.
- **Auth / deployment / transport.** Official: remote HTTPS, a **scoped Bearer access token** (a long-lived token you mint with a specific scope — *not* an interactive OAuth flow, so there is no refresh/expiry dance), streamable-HTTP/SSE — nothing to run locally, but nothing runs *without* the network and the host must speak remote-HTTP MCP. This server: local stdio (`node dist/index.js`), long-lived API token, runs in-process next to the host. **Local-filesystem attachment confinement (`FASTMAIL_ATTACH_DIR`) is only meaningful for a local server; it is structurally impossible remotely** (code-backed — the official server has no filesystem on your machine to confine to, and no download tool at all).
- **Host-support precondition (unverified).** Switching to *or* adding the official server assumes your MCP host can natively mount a remote Bearer-auth HTTP server. Today the host runs this server as a stdio `command: node` launcher. The probe proves the **endpoint** is reachable from a script; it does **not** prove the **host** can wire to it. One config question, not a known blocker — but price an adapter/bridge into any migration estimate until confirmed.
- **Host-capability dependency (widget UX).** The official server advertises an `extensions` capability and ships MCP-UI widgets (`compose_event`, `confirm-delete-event/note/contact`, an `email-view`). On a host that renders MCP-UI these are a real UX win; on a plain stdio CLI host they degrade to "staged, awaiting a confirm the host can't show." A **host-capability dependency**, not a portable feature — and whether it matters at all depends on your client.

## 3. What each side does better

**This server does better:**
- **`raw:true` escape hatch on every email tool** (code-backed) — full canonical JMAP on demand. The official server has **no raw mode**; what it returns is what you get. This matters whenever simplification drops a field you actually needed.
- **Local attachment download + path confinement** (code-backed) — `get_email_attachments` + `download_attachment` (to disk, confined to `FASTMAIL_ATTACH_DIR` when set). The official server **cannot download an attachment at all** (schema-inferred — upload only).
- **`search_emails` structured fields** (code-backed) — multi-criteria JMAP-level search you compose from **fields** (from/to/cc/bcc/subject/date/mailbox, alongside the free-text query), no query-string to build. (This was a separate `advanced_search` tool; it is now consolidated into `search_emails`.) The official `search_email` is a **Gmail-style string** (schema-confirmed) — and it's *not* qualifier-poor: its schema documents `from to cc bcc with subject body text in before after is has` plus extensions (`me`, plain-name contact lookup, `in:` by id/name/role, relative dates `7d`/`1w`). So this server's edge here is **programmatic/structured composition** (machine-friendly, no string parsing), *not* a larger searchable surface — the two are different input models for comparable expressiveness.
- **Explicit bulk-by-id tools** (code-backed) — `bulk_move`/`bulk_pin`/`bulk_mark_read`/`bulk_delete`/`bulk_add_labels`/`bulk_remove_labels`. The official server batches via `ids[]` on `update_email`/`archive_email`/`delete_email` (schema-inferred) but has no single-call "pin these 30" / "move these to mailbox X".
- **Full draft lifecycle** (code-backed) — `create_draft` → `edit_draft` (partial edits, `clearFields`, attachment add/remove, faithful destroy-and-recreate that preserves threading headers + keywords) → `send_draft`. The official can only *create* a draft (`draft_email`); it has **no edit-draft and no send-saved-draft tool**, and its `update_email` schema exposes no body/recipient fields (schema-inferred — not executed; a runtime probe could in principle reveal undeclared field acceptance, which is why the §1 write-probe caveat applies) — so any compose-iteratively-then-send flow is this-server-only on the published surface. *(Symmetric caveat: this server's own draft-edit/send behaviour is code-backed — read from source this session, not executed — so this row is two unexecuted write paths compared, not an observed head-to-head.)*
- **Reply quoting + a draft-safe default** (code-backed) — `reply_email` sanitizes the quoted HTML (strips scripts/handlers, keeps formatting), attributes and top-posts to match the web client, and **defaults to saving a draft** (`send=false`) rather than transmitting. Plus lenient recipient input (comma-separated strings, `"Name <email>"`, bool-or-string) vs the official's structured `[{name,email}]`.
- **Runs in the user's host today** (code-backed) — stdio, already wired, no transport question.

**Official does better:**
- **Token efficiency on the core read path *as each ships today*** (probe-backed) — this is the official's one *measured* edge, and it is real: its default output is leaner than this server's current simplified output for list, single-email, and thread (numbers in §Appendix). The gap is mostly this server's pretty-printing + extra fields — compacting serialization, and (if chosen) trimming some default fields, would erase it (§5 P0/P1) — but **today this server is the heavier one**. It's small in absolute terms, and it's the only place the comparison rests on a live measurement rather than a schema reading; weigh it as a modest, currently-real point in the official's favour, not as decisive.
- **Reads thread bodies in one call** (probe-backed) — `read_thread` returns quote-stripped bodies inline; this server's `get_thread` returns metadata only, forcing a `get_thread` + `get_email`×N fan-out to read a conversation. (This is the durable thread advantage — a round-trip/capability difference, not the single-thread token number, which is illustrative only; see §Appendix.)
- **Expands recurring calendar events, and returns richer event detail inline** (probe-backed) — over the same window, `search_events` returned each *occurrence* of a recurring event as its own dated row (e.g. a weekly session appeared 7× with a `recurrenceId`), so "what's on next week" is directly answerable; this server's `list_calendar_events` returned one row per event object with **no occurrence expansion** and a sparse field set (`start`/`end`/`title`/`id`/`url`), pushing both the recurrence math and the detail (participants, location, duration, timezone) onto a per-event `get_calendar_event` fan-out. The official's event rows also carry that detail inline. This is a genuine read-side capability gap, not a token artifact.
- **Strips quoted reply history on read by default** (official schema-inferred; this server code-backed) — `read_email` takes `includeQuotedText` (default false), returning just the new message and dropping the quoted chain unless asked; this server's `get_email` always returns the full `bodyText`. The three sampled messages carried no quoted chain (their bodies were byte-identical across both servers), so they cannot themselves probe this; that the edge "only bites on long reply chains" is reasoned from the feature's definition, not measured here — but for a reply-heavy mailbox it is a real cleanliness/cost edge.
- **Notes** (schema-inferred) — `search_notes`/`read_note` (+ write tools). This server has **zero** notes support.
- **Memos** (schema-inferred) — private per-email annotations via `set_memo`, surfaced inline in `read_email`/`search_email`. This server has none — this is both a write gap *and* a read gap.
- **Contacts write** (schema-inferred) — `create_contact`/`update_contact`/`delete_contact`. This server's contacts are **read-only** (list/get/search only).
- **Calendar invitations / RSVP** (schema-inferred) — `create_event`/`update_event` explicitly send iMIP to attendees, and `rsvp_event` responds to invitations. This server can attach participants to an event (code-backed) but does not surface iMIP-sending or any RSVP path.
- **Forward** (schema-inferred) — `draft_email`/`send_email` support `mode: 'forward'`. This server has reply but **no forward**.
- **Unified, markdown compose** (schema-inferred) — one `draft_email`/`send_email` pair driven by `mode: new|reply|forward`, with a **markdown `body`** auto-converted to HTML (+ a plain-text alternative) and `replyAll`. Ergonomically simpler for a caller than this server's separate compose tools and explicit `htmlBody`/`textBody` (which has no markdown helper — formatted mail means writing HTML yourself). A cleaner *interface*, even though this server's draft *lifecycle* is richer.
- **Gmail-style search string** (schema-confirmed) — friendlier for ad-hoc natural queries than building structured criteria, with a rich qualifier set + extensions (relative dates `7d`/`1w`, plain-name contact lookup, `me`), and it **surfaces any attached memo inline** in the results. A usability edge, not a capability this server lacks an answer to.
- **Richer label model** (probe-backed) — `list_labels` returns id/path/role/parentId/counts; this server's `list_mailboxes` is lighter but carries less structure.
- **Trajectory** *(judgment)* — the capability gaps below are a snapshot, not a fixed list. A first-party v1.0.0 that already matches this server's body-trimming and ships Notes/memos/contacts-write out of the gate will keep adding; the port shortlist is a gap that **re-opens** as the official server iterates.

## 4. Gaps each way

**This server only (official lacks entirely):** attachment **download to disk** (the official has no download tool — though it *does* surface attachment metadata inline on read, so attachment *listing* is shared); local-filesystem **confinement**; `raw:true` full-JMAP; `search_emails`' structured field composition; **draft editing** (`edit_draft`) and **sending a saved draft** (`send_draft`) — the official can create a draft but, per its schema, never modify or later-send it (schema-inferred — its `update_email` exposes no body/recipient fields, and was not executed this session); `get_account_summary`; explicit bulk-by-id pin/move.

**Official-only (this server lacks entirely):** **Notes** (read + write); **memos** (read + write); **contacts write**; **forward**; **RSVP** to invitations; **iMIP-sending** calendar invites (surfaced as such); dedicated **`archive_email`**; **Gmail-style** search dialect; interactive **widget UX**.

**Load-bearing asymmetry (the headline, framed honestly):** as each server ships today, this server's *simplified* output is **heavier** than the official's default — but this is a tradeoff, not pure waste. Two components: (a) this server **pretty-prints** (~17% pure-whitespace overhead by bytes, no information — a clean, free loss), and (b) this server **returns more fields** in its default view (`blobId`, `listUnsubscribe`, `references`, `inReplyTo`, `keywords`, `replyTo`, `size`). Component (b) is *paid-for content*, not inefficiency: whether those fields belong in the default view is a **product judgment** (a list reader rarely needs threading headers or a blob id), not a measured defect. Drop the pretty-printing and move those fields behind `raw:true` and this server's per-message weight falls *below* the official's (204 vs 257 tok; these per-message means are over the 24-of-25 cross-server message join, so treat them as indicative — see A1). Most of that lead is **information-preserving**: this server omits empty `cc`/`bcc` and false flags that the official emits regardless (omit-empties — same information, fewer tokens). The trimmed view genuinely carries *less* on only two axes — no `isAnswered`/`isForwarded`, and name-only `mailboxes` vs structured `labels` (§Appendix). Bottom line: this server ships **heavier today**, can be made **modestly lighter at near-equal information**, and either way **token weight is not a meaningful differentiator** — only the pretty-print removal is an unambiguous free gain.

## 5. Recommendation paths + port shortlist

Four ways to go (§1), all defensible depending on what you value:
- **Develop this server** — keep using it and add what you want. The shortlist below ranks candidate additions (all *candidates*, none decided): the cheap ones (P0, P2) are low-stakes, P1 is an open per-field decision (it changes the default output), and the expensive ones are where you should weigh "build it" against "just get it from the official server."
- **Switch to the official** — if you'd rather not maintain anything and want its exclusive features now. Gate a *final* switch behind a consented mutation probe (its write side is schema-inferred, not tested) and the host-wiring check (§2).
- **Run both** — this server for download/`raw`/structured `search_emails`/bulk, the official for its exclusives (Notes/memos/contacts-write/forward/RSVP). **Cost:** two overlapping tool sets and a **search-dialect-confusion hazard** — this server's `search_emails` takes free-text plus JMAP-backed structured fields while the official `search_email` takes a Gmail-style string, so with both loaded a wrong-dialect query fails silently or misleads. Loading this server **on-demand** (only when you need a download) sidesteps that cost, since the two are never live at once.
- **Federate into one surface (a solid middle ground)** — keep this server as the single front door: it serves its own strengths **natively** (drafts, attachment download, `raw:true`, structured `search_emails`, bulk) and, instead of *reimplementing* the official's exclusives, **forwards those calls to the official MCP** and re-exports their tools as its own. You get the first-party Notes/memos/contacts-write/RSVP with **no business logic to build or maintain**, while keeping the faithful native draft + attachment paths (no lossy round-trip through the official's plaintext-only read). MCP-server-as-MCP-client is a standard pattern, and the official-side client already exists (the read-only HTTP/SSE client built for this audit). Auth is just a **second scoped Bearer access token** — the same kind already used here, long-lived, **no OAuth refresh dance**. **Curate to avoid overlap:** proxy only the exclusives (and optionally the official-better reads — forward, one-call thread bodies, calendar-recurrence), keep everything you do as-well natively — which *removes* the Run-both search-dialect hazard, since each capability is exposed exactly once. **Caveats:** the interactive widget tools (`compose_event` / confirm-delete) likely won't survive a stdio proxy hop — skip them; proxied official *mutators* belong behind a deliberate allowlist (same posture as this audit's read-only gate); and one surface now mixes local-fast and remote tools, so a remote outage degrades the proxied half. **Bonus:** a local stdio server bridging to the official's HTTP endpoint *is* the adapter the §2 host-support precondition asks for — so this doubles as the way to use the official from a stdio-only host. Net effect: it converts the shortlist's most expensive items (Notes / memos / contacts-write) from "build" to "forward."

**Port shortlist (for the develop path).** Value = coarse high/med/low of how often the gap bites a daily Fastmail user *(a judgment — override freely)*. JMAP-reachability is **inferred, not probe-verified** (the audit ran no mutators): "likely" = reuses machinery this server already has; "Verify" = depends on a mechanism not yet confirmed. Un-portable items are dropped (they're reasons to run the official alongside). Below the P2 line, "build it here" competes with "just use the official for that piece" — decide per feature. **Rows tagged 🔁 are strong Option-4 *proxy* candidates** — forwarding to the official's first-party implementation likely beats building and maintaining your own, *especially* the Verify-gated ones whose JMAP reachability is unconfirmed.

¹ **P0 and P1 are different in kind, and neither is about "winning" on tokens** (token weight is not a differentiator — §1). **P0** (compact serialization) is **information-preserving** — it drops only whitespace, so it's low-stakes and reversible, though still the operator's call. **P1 is NOT free**: it would **remove real fields from the default output**, which is a user-visible contract change and a per-field product decision — *explicitly not decided here*. Some of those fields may well belong in the default. Listed top of the shortlist only because they're *cheap to implement*, not because either is a settled action.

**Dropped as un-portable (no public-JMAP analog to *build*):** the **widget UX** (host-capability dependency). Note that "un-portable" now means "can't be *rebuilt* on JMAP" — it no longer means "must run a second server," because **Option 4 lets you forward to the official's implementation instead of porting it.** The widget tools are the one thing Option 4 also can't cleanly deliver (they likely won't survive a stdio proxy hop), so genuinely-interactive widget UX remains the only capability that truly requires the official server running in a widget-capable host. If, after verification, **Notes** turns out not to be public-JMAP-reachable, that is now a reason to *proxy* them (Option 4), not necessarily to run two separate servers.

## Appendix

### A1. Probe method (abstract)
Read-only only, both servers, behind a single frozen-allowlist choke-point (default-deny; ignores the server's `readOnlyHint`, since four official tools self-report read-only while staging mutations). Official reached via a purpose-built HTTP/SSE client (token from env, no body/header logging); this server reached via its stdio harness wrapped in the same allowlist. Token counts use a local `gpt-tokenizer` o200k_base tokenizer (v2.9.0) — a **GPT-family proxy**, since the host model's tokenizer is not public; **raw bytes are co-reported as the tokenizer-independent measure and every directional claim holds in bytes too**. Per-message figures normalize over the **common Message-ID set** (for the 24 messages present on *both* servers' pages, cross-server JMAP ids were equal — 0 mismatches — so a direct id join is valid; the 1 non-joiner is simply absent from one side's page, not an id-mismatch). That 1 of 25 is *attributed* to **page-boundary differences** (each server's "25 most recent" set is not the identical 25, due to ordering/inclusion at the edge) — a mechanism argument, not a measured no-skew check; with 24 of 25 retained, treat the per-message averages as indicative rather than precise. Note the two A2 tables use different bases: the as-served page is **25 messages plus the envelope**, while per-message averages are over the **24-message common set**, so the two do not multiply directly (e.g. official 257×24 ≠ the 6,375 as-served figure). Information is held constant: an omitted field is a content delta, never counted as a token "win." The negative-control test confirmed the gate blocks all forbidden tools pre-request (8/8 official incl. all four widget liars, 8/8 this server) with zero requests built.

### A2. Token weights (probe-backed, 2026-06-25; o200k_base proxy + bytes)

**Email list page (25 recent), as-served (what the host receives):**

| Representation | tokens | bytes | serialization |
|---|---|---|---|
| this server (simplified) | 9,605 | 24,962 | pretty-printed |
| this server (`raw:true`) | 11,481 | 31,494 | pretty-printed |
| **official** | **6,375** | **18,338** | compact |

The as-served gap (this server 9,605 → official 6,375 tok; in bytes 24,962 → 18,338) has two parts:
- **Pretty-print + envelope whitespace** — re-serializing the *identical* content compactly gives 9,605 → 8,031 tok and 24,962 → 20,675 bytes, i.e. **~4,287 bytes (~17% of the as-served pretty-printed payload) / ~1,574 tokens (~16%)** of pure whitespace, zero information. **The byte figure is the reliable one**: whitespace tokenization is exactly where a GPT-family proxy and the host model diverge most, so trust the ~17% byte reduction over the ~16% token count here. This part loses zero information (it's only whitespace), but it does **not** by itself close the per-message gap (compact-whole this server 8,031 still > official 6,375).
- **Remaining difference** = this server carrying more fields (next table).

**Per-message normalized (compact re-serialization, common set n=24):**

| | tokens/msg | bytes/msg | field richness |
|---|---|---|---|
| this server (as shipped) | 322 | 830 | its field set + 7 extra |
| **official** | **257** | **737** | official field set |
| this server (**projected**: 7 extra fields moved behind `raw:true`) | **204** | **562** | near-equal info (see below) |

The projected 204/562 is a **direct measurement** (re-serialize each common-set message with exactly the seven fields removed, then count — removing a field also removes its key + delimiters, so the realized saving exceeds the field-value subtotal below). Crucially, most of this server's resulting lead is **information-preserving**, not "carrying less": this server **omits empty/false fields** (`addIf` drops empty arrays; `addFlag` drops `isFlagged`/`isDraft`/`isReply`/`hasAttachment` when false — `email-formatter.ts`), so on these mostly-no-`cc`, unflagged sample messages this server spends no tokens on `cc`/`bcc`/flags while the official emits them anyway (e.g. `cc:null`). That is a genuine efficiency at equal information. This server carries *less* on only two points: it surfaces no `isAnswered`/`isForwarded` at all, and its `mailboxes` are name-only vs the official's structured `labels`. So this server's projected view is **modestly lighter at near-equal information** — not a strict-superset win, but not "lighter by dropping content" either. Combined with the as-shipped reality (this server is heavier *today*), the fair takeaway is **token weight is not a meaningful differentiator**; pretty-print removal is the only unambiguous free gain, and omit-empties is a real but modest edge for this server.

**Per-field token *value* drivers over the common 24 (indicative only).** These are field values tokenized in isolation; o200k_base merges across boundaries, so per-field token counts are **non-additive and do not sum to the per-message totals** — read them as a directional decomposition, and lean on bytes. (A per-field structural/key breakdown is omitted deliberately: it could only be a back-solved residual, which is not a measurement.)

| field | this server (value tok) | official (value tok) |
|---|---|---|
| preview | 1,534 | 1,534 *(identical — info held constant)* |
| messageId | 781 | 781 |
| from | 317 | 490 *(official structured `{name,email}`)* |
| to | 278 | 425 |
| labels / mailboxes | 88 *(names)* | 566 *(full `{id,name,path,role}`)* |

Reading: this server's net heaviness is the seven extra content fields it returns; the official is heavier on structured `from`/`to`/`labels` *and* on always emitting empty `cc`/`bcc`/status-flags that this server omits. Neither is a strict superset, so there is no single clean iso-information number — but the omit-empties difference favors this server at equal information (see the projected-row note above).

**Single email (`get_email` vs `read_email`), n=3 — per-sample observations, not a stable rate:**

| sample | this server `get_email` | official `read_email` | this server `raw:true` | body chars (this server / official) |
|---|---|---|---|---|
| #0 | 756 | 432 | 2,525 | 543 / 543 |
| #1 | 970 | 652 | 2,879 | 1,447 / 1,447 |
| #2 | 505 | 308 | 928 | 472 / 472 |

Three samples spanning a ~3× size range; the official is lighter in each, by the same *kind* of metadata overhead seen in the list — though the per-sample delta varies (324 / 318 / 197 tok), so it isn't a single constant — and n=3 cannot establish a stable per-email rate, so read these as individual observations. Both return **byte-identical** plain-text bodies (HTML omitted on both sides; these samples carried no quoted reply chain, so the official's default quote-stripping made no difference here), so the gap is metadata, not body content. A separate targeted probe confirms the official's read path is **text-only even for richly-HTML mail**: on a message this server reports as ~57 KB of HTML, `read_email` returned a plain-text rendering with **no HTML markup in any field and no `blobId`** — so unlike this server (whose `raw:true` recovers the true `htmlBody`), the official exposes **no path to a message's HTML source at all**. That's why a *federated read* off the official (§5, Option 4) could never recover formatting — it's a structural limit of their read surface, not a sampling artifact. `raw:true` (full JMAP incl. HTML) is ~2–3× heavier on these short-bodied samples — confirming simplification is a real win *over raw JMAP*, which the official server matches independently. (The earlier draft's larger multiples came from a different, HTML-heavy sample set; the multiple scales with how much HTML the message carries.)

**Thread (`get_thread` vs `read_thread`) — n=1, illustrative only:** this server 1,150 tok (4 members, **no bodies**); official 966 tok (5 members, **with** quote-stripped bodies). (Byte counts were not separately captured for this single thread; with n=1 and the caveat below, the token figure is illustrative only and no directional claim rests on it.) This server being *heavier* while carrying *fewer* members and *no* bodies is the same per-member metadata + pretty-print overhead measured in the list section, not a thread-specific effect — and this thread had short bodies, so **do not generalize the token number** (a thread with long un-quotable bodies could invert it). The durable difference is the **capability**: the official reads a thread's content in one call; this server needs `get_thread` + `get_email`×N. The member-count gap (4 vs 5) is **not a bug**: this server's `getThread(..., includeDrafts = false)` deliberately excludes draft messages (the missing member was a trashed draft), while the official includes them — a behavioral difference that is defensible but **can surprise** a user who expects the web client's view: a thread made *entirely* of trashed drafts comes back from this server with **0 members** while the official returns them. Worth knowing, and arguably an argument for a future `includeDrafts` toggle.
